# 기술 설계 및 스택 선정 보고서

본 보고서는 이벤트 로그 파이프라인 구축에 있어 채택된 기술 스택의 선정 이유와 설계 의도를 설명합니다.

## 1. 데이터베이스 선정: PostgreSQL vs NoSQL

| 분석 항목 | PostgreSQL (선택) | Elasticsearch | MongoDB |
| :--- | :--- | :--- | :--- |
| **분석 정확도** | ACID 준수로 매우 높음 | 검색 중심, 통계 정밀도 다소 낮음 | 준수하나 JOIN 분석 시 취약 |
| **관계형 연산** | **JOIN 연산 최적화** | JOIN 거의 불가능 (매우 어려움) | 가능하지만 대량 집계 시 성능 저하 |
| **유연성** | **JSONB를 통한 하이브리드 지원** | 매우 높음 (Schema-less) | 매우 높음 (Document) |

### 선정 사유:
우리의 핵심 분석 목표인 **"조회 상품 vs 구매 상품 비교"**는 상품 마스터 데이터와 이벤트 로그 간의 강력한 **JOIN 연산**이 필수적입니다. NoSQL(ES, Mongo)은 이 부분에서 개발 리소스가 많이 들거나 성능이 떨어지는 반면, PostgreSQL은 SQL의 강력함과 JSONB의 유연성을 동시에 가져갈 수 있어 가장 효율적인 대안이었습니다.

## 2. 시각화 도구 선정: Streamlit

### 선정 사유:
단순히 차트만 보여주는 것이 아니라, **"카테고리 필터링 조회 기능"**을 구현하기 위해 Python 기반의 Streamlit을 선택했습니다. 
- **커스텀 UI**: 평가자가 직접 카테고리를 바꿔가며 분석 결과를 확인할 수 있는 인터랙티브한 환경 제공.
- **통합성**: 데이터 생성기(`generator.py`)와 동일한 Python 환경에서 돌아가므로 관리 및 실행이 용이함.

## 3. 스키마 설계 의도: Hybrid Strategy
- **정형 데이터(Standard Columns)**: ID, 시각, 타입 등은 고정 컬럼으로 두어 검색 속도와 데이터 무결성 보장.
- **비정형 데이터(JSONB)**: 검색어, 할인율 등 이벤트마다 달라지는Context 정보는 JSONB에 담아 추후 로직 변경 시 DB 스키마 수정 없이 즉각 대응 가능.

---
이와 같은 설계는 현대적인 데이터 엔지니어링에서 **"안정적인 운영과 유연한 분석"** 두 마리 토끼를 잡기 위해 널리 사용되는 전략입니다.
